# ЧИСТОВОЙ ПЛАН ПРОЕКТА
## Локальная desktop-система управления охраной труда на предприятии

**Статус документа:** final v1  
**Назначение:** основной технический документ для проектирования, реализации, проверки архитектуры и AI-разработки  
**Тип документа:** product + architecture + implementation constraints

---

## 1. ЦЕЛЬ ПРОЕКТА

Создать локальную desktop-систему для инспектора по охране труда, которая:

1. ведёт полный учёт по охране труда;
2. контролирует сроки, допуски, просрочки и критические нарушения;
3. формирует управленческую картину по предприятию;
4. показывает изменения нормативно-правовых актов и профильные новости из доверенных источников;
5. снижает риск пропуска обязательных действий;
6. поддерживает перенос данных, резервное копирование и восстановление после сбоев.

**Предметная область:** промышленное предприятие.  
**Охват:** весь персонал в зоне ответственности инспектора по ОТ.  
**Целевая модель эксплуатации:** локальная установка, основной оператор один.

---

## 2. ГРАНИЦЫ ПРОДУКТА

### 2.1. Что это за продукт
Это локальная система инспектора по охране труда с:

- локальной БД;
- локальным интерфейсом;
- внешним контуром чтения новостей и НПА;
- внешним почтовым контуром для отправки отчётов;
- опциональным внешним путём хранения резервных копий.

### 2.2. Что это не за продукт
Это не:

- web-first платформа;
- ERP;
- кадровая система общего назначения;
- медицинская информационная система;
- обязательный облачный SaaS;
- система, требующая обязательной локальной сети предприятия;
- real-time multi-user система.

### 2.3. Коммерческая модель
Продукт поставляется по модели:

`установил -> настроил -> работаешь`

Приоритеты:

- простая установка;
- отсутствие обязательного сервера;
- отсутствие обязательной корпоративной сети;
- минимальная зависимость от внешней ИТ-поддержки;
- предсказуемое локальное поведение.

## 3. БАЗОВАЯ МОДЕЛЬ ИСПОЛЬЗОВАНИЯ

### 3.1. Основной пользователь
Основной оператор системы: **инспектор по охране труда**.

### 3.2. Внешние участники процесса
Вне программы участвуют:

- HR, как источник части входных данных;
- руководитель, как получатель отчётов и пользователь режима просмотра;
- начальники участков, как внешние участники бизнес-процесса без полноценной учётной роли в первой версии.

### 3.3. Эксплуатационная модель
Система не рассчитывается на конкурентную многопользовательскую работу как базовый режим.

Базовый сценарий:

- один основной оператор;
- данные ведутся локально;
- внешние участники получают результаты через отчёты, экспорт, печатные формы и режим просмотра.

## 4. ЦЕЛИ И ПРИОРИТЕТЫ

### 4.1. Фиксированный баланс целей
Обязательные контуры:

1. полный учёт по ОТ;
2. контроль сроков, просрочек и допусков;

### 4.2. Приоритет бизнес-функций

## 5. АРХИТЕКТУРНЫЕ ПРИНЦИПЫ

### 5.1. Общий стиль
Архитектура системы: **модульная desktop-архитектура**.

Основные требования:

- низкая связанность модулей;
- явные интерфейсы между модулями;
- отсутствие бизнес-логики в UI;
- отсутствие скрытых зависимостей между доменными блоками;
- отказ от “быстрых” решений, нарушающих целостность модели.

### 5.2. Ключевые архитектурные принципы
1. UI только отображает состояние и инициирует действия.
2. Оркестратор координирует сценарии, но не должен становиться хранилищем всей бизнес-логики.
3. Доменные правила живут в профильных модулях.
4. Все критичные изменения выполняются через единый application-flow.
5. Внешний контур не имеет права напрямую изменять внутренние данные предприятия.
6. Любая операция, способная повлиять на целостность данных, должна быть логируемой и тестируемой.
7. Долгие операции не должны блокировать критичный пользовательский сценарий без явного статуса выполнения.
8. Архивные данные не переиспользуются как новые активные сущности.

### 5.3. Требование к модульности кода
Каждая самодостаточная функция или use-case-операция должна быть выделена в отдельный файл соответствующего модуля.  
Не допускается бессистемное объединение независимых сценариев в один файл “для удобства”.

Все функции должны иметь понятное описание:

- на русском;
- на английском.

## 6. ВЫСОКОУРОВНЕВАЯ АРХИТЕКТУРА

### 6.1. Слои системы
Система должна быть разделена минимум на следующие слои:

- `UI layer`
- `Application / Orchestration layer`
- `Domain layer`
- `Infrastructure layer`
- `Persistence layer`

### 6.2. Ответственность слоёв

#### UI layer
Отвечает за:

- экраны;
- отображение;
- ввод пользователя;
- запуск команд;
- отображение ошибок и состояний операций.

Не отвечает за:

- принятие бизнес-решений;
- расчёт статусов;
- доступ к БД напрямую;
- пересчёт правил допуска.

#### Application / Orchestration layer
Отвечает за:

- запуск пользовательских сценариев;
- координацию нескольких модулей в рамках одного use case;
- управление транзакционными границами;
- запуск пересчёта;
- запуск фоновых операций;
- публикацию доменных событий верхнего уровня.

Не отвечает за:

- хранение доменных правил всех модулей внутри себя;
- логику конкретного типа инструктажа, СИЗ, медицины или наряда.

#### Domain layer
Отвечает за:

- правила предметной области;
- статусные вычисления;
- валидацию инвариантов;
- правила критичности;
- интерпретацию обязательности и актуальности.

#### Infrastructure layer
Отвечает за:

- файловую систему;
- почту;
- внешние источники НПА;
- генерацию резервных копий;
- адаптеры импорта/экспорта;
- логирование;
- криптографические и системные сервисы.

#### Persistence layer
Отвечает за:

- локальную БД;
- миграции;
- репозитории;
- изолированную работу с транзакциями и запросами.

## 7. ОРКЕСТРАТОР

### 7.1. Обязательность
Оркестратор обязателен как application-координатор.

### 7.2. Его роль
Оркестратор должен:

- принимать команды из UI;
- вызывать нужные доменные и инфраструктурные модули;
- обеспечивать правильный порядок шагов;
- управлять жизненным циклом операции;
- инициировать пересчёт статусов;
- инициировать формирование уведомлений;
- инициировать фоновую обработку долгих задач;
- централизованно завершать сценарий успехом или ошибкой.

### 7.3. Что запрещено оркестратору
Оркестратор не должен:

- содержать всю доменную логику внутри себя;
- напрямую хранить состояние UI;
- становиться универсальным “сервисом всего”;
- дублировать правила доменных модулей;
- выполнять прямые SQL-операции в обход слоя хранения.

### 7.4. Архитектурное правило
Любой пользовательский сценарий, который затрагивает несколько модулей или меняет критичные данные, должен идти через оркестратор.

Содержит:

- опциональный внешний путь хранения бэкапов.

### 8.3. Жёсткое правило изоляции
Внешний контур может только:

- кэшировать;

Внешний контур не может:

- напрямую изменять сотрудников;
- напрямую изменять статусные данные;
- создавать или редактировать боевые записи внутреннего контура без явного сценария, подтверждённого через оркестратор.

- архив сотрудников;

### 9.3. Модуль контроля по ОТ
Отвечает за:

- расчёт итогового статуса сотрудника;
- выявление нарушений;
- правила критичности;

Отвечает за:

- снятие уведомлений по факту устранения причины;

- статистические сводки;
- управленческие отчёты;

Отвечает только за:

- отправку подготовленных сообщений;
- повторные попытки;
- фиксацию ошибок доставки.

Не владеет логикой формирования отчёта и не редактирует основную БД, кроме записи технического статуса отправки через допустимый интерфейс.

### 9.7. Модуль импорта
Отвечает за:

- чтение поддерживаемых форматов;
- распознавание;
- сопоставление входных данных;
- формирование черновиков;
- передачу на проверку;
- применение подтверждённых изменений через оркестратор.

### 9.8. Модуль экспорта
Отвечает за:

- полный экспорт состояния;
- перенос между установками;
- резервный экспорт;
- версионирование экспортного формата.

### 9.9. Модуль резервного копирования и восстановления

- ручной бэкап;
- автоматический бэкап;
- хранение копий;
- проверку целостности;
- восстановление;
- страховочную копию перед восстановлением.

### 9.10. Модуль новостей / НПА
Отвечает за:

- чтение доверенных источников;
- кэширование;
- дедупликацию материалов;
- хранение ссылок и статусов прочтения;
- сигнализацию о новых или изменённых материалах.

### 9.11. Модуль настроек
Отвечает за:

- параметры почты;
- параметры бэкапа;
- параметры импорта/экспорта;
- список доверенных источников;
- параметры безопасности;
- параметры визуального поведения;
- системные пороги предупреждений.

### 9.12. Модуль шаблонов и документов
Поздний контур.

На первой рабочей версии:

- архитектурно предусмотрен;
- полноценно не реализуется;
- не должен влиять на основные контуры.

## 11. МОДЕЛЬ КОНСИСТЕНТНОСТИ ДАННЫХ

### 11.1. Источник истины
Источником истины являются доменные записи в основной БД.

### 11.2. Вычисляемые статусы
Итоговые статусы не должны редактироваться вручную как первичные данные.  
Они должны:

- вычисляться на основании доменных сущностей;
- пересчитываться при изменении связанных данных;
- быть воспроизводимыми детерминированно.

### 11.3. Пересчёт
Система должна поддерживать:

- локальный пересчёт при изменении конкретной сущности;
- пакетный пересчёт после импорта;
- полный пересчёт для сервисных сценариев восстановления или диагностики.

### 11.4. Правило завершения операции
Операция, меняющая предметные данные, считается завершённой только если:

1. изменения записаны;
2. связанные статусы пересчитаны;
3. уведомления синхронизированы;
4. критичное событие залогировано.

Если фаза не завершилась, операция не считается успешно применённой.

## 13. ДОМЕННЫЕ ИНВАРИАНТЫ

### 13.1. Общие
1. Архивная карточка сотрудника не переиспользуется под нового человека.
2. Итоговый статус сотрудника определяется автоматически.
3. Внешний источник не может изменить внутренние боевые данные напрямую.
4. Закрытый наряд-допуск не редактируется.
5. Импорт не пишет напрямую в боевые сущности без стадии черновика и подтверждения.
6. Пароль, recovery-код и сервисный код никогда не хранятся и не логируются в открытом виде.

### 13.2. По сотрудникам
1. У сотрудника одна основная должность.
2. Новый сотрудник создаётся новой карточкой.
3. Архивность сотрудника не должна уничтожать историю его событий.

### 13.3. По статусам
1. Частные статусы рассчитываются независимо по каждому доменному контуру.
2. Итоговый статус сотрудника вычисляется по самому тяжёлому релевантному нарушению.
3. Ручное редактирование итогового статуса запрещено.

### 13.4. По нарядам-допускам
1. Истечение по времени не равно корректному закрытию.
2. Закрытый наряд только хранится в истории.
3. Если работа требует активного наряда, отсутствие актуального наряда считается критичным.

### 13.5. По восстановлению
1. Перед восстановлением всегда создаётся страховочная копия текущего состояния.
2. Восстановление не должно начинаться без явного подтверждения пользователя.
3. После восстановления должна быть выполнена проверка целостности и сервисный пересчёт.

## 18. СИЗ

### 18.1. Базовая логика
Система должна знать:

- количество;

### 18.2. Определение нормы

### 18.3. Статусы
- актуально
- внимание
- критично: срок истёк
- критично: обязательный СИЗ не выдан
- несоответствие норме

### 18.4. Порог предупреждения
Стартовое значение: 7 дней.  
Должно настраиваться.

### 18.5. Ограничение первой версии
Массовая выдача СИЗ в первой версии не реализуется.

## 24. БЭКАП И ВОССТАНОВЛЕНИЕ

### 24.1. Обязательные функции
- локальный авто-бэкап;
- ручной бэкап;
- ручной экспорт;
- восстановление.

### 24.2. Опциональный облачный путь
Допускается как внешний сценарий под ответственность предприятия.

### 24.3. Правила бэкапа
Бэкап должен:

- создаваться как атомарная копия согласованного состояния;
- иметь метаданные версии;
- иметь проверку успешности;
- не считаться валидным без завершения процедуры.

### 24.4. Правила восстановления
Перед восстановлением система обязана:

1. создать страховочную копию текущего состояния;
2. показать предупреждение;
3. запросить подтверждение;
4. выполнить восстановление;
5. проверить целостность;
6. выполнить сервисный пересчёт зависимых статусов;
7. зафиксировать событие в audit-логе.

### 24.5. Экран восстановления
Две отдельные вкладки:

- Резервные копии
- Восстановление

### 24.6. Версионирование формата
Форматы бэкапа и экспорта должны иметь явную версию, чтобы в будущем поддерживать миграции и совместимость.

## 25. ОТЧЁТЫ И ПОЧТА

### 25.1. Ежедневный отчёт руководителю
Обязательная функция, но отключаемая.

### 25.2. Почтовый модуль
Почта:

- получает готовый отчёт;
- отправляет;
- повторяет попытки;
- сообщает об ошибке.

### 25.3. Ошибки отправки
- 3 попытки;
- пауза между попытками;
- уведомление при неудаче;
- возможность подготовить отчёт для ручной отправки.

### 25.4. Архитектурное правило
Почтовый модуль не должен принимать решений о содержимом отчёта и не должен владеть логикой допустимости отправки.

## 26. БЕЗОПАСНОСТЬ

### 26.1. Два рубежа доступа
- вход в ОС;
- вход в программу.

### 26.2. Роли
**Инспектор**
- полный рабочий доступ.

**Руководитель**
- только просмотр;
- без редактирования;
- без импорта;
- без экспорта;
- без восстановления;
- без смены настроек;
- без смены пароля.

### 26.3. Пароль программы
- хранится не в открытом виде;
- не отправляется;
- не логируется;
- после серии неудачных попыток вход временно блокируется.

### 26.4. Восстановление доступа
Не использовать универсальный мастер-пароль разработчика.

Разрешённые механизмы:

- recovery-код пользователя;
- recovery-файл;
- разовый сервисный код по ID установки.

### 26.5. Recovery-файл
Содержит:

- ID установки;
- recovery-код;
- дату создания;
- предупреждение хранить отдельно.

Не содержит пароль.

### 26.6. Аварийное восстановление
Если забыты и пароль, и recovery-код:

- на экране входа показывается ID установки;
- пользователь обращается к разработчику;
- разработчик выдаёт разовый сервисный код;
- код открывает только режим сброса;
- после сброса генерируется новый recovery-код и новый recovery-файл.

### 26.7. Требования к сервисному коду
Сервисный код должен быть:

- одноразовым;
- привязанным к установке;
- ограниченным по времени или числу применений;
- пригодным только для входа в режим сброса;
- не открывающим рабочий интерфейс;
- логируемым как критичное событие безопасности.

### 26.8. Бэкап и экспорт
В первой версии допускаются как обычные файлы без дополнительного шифрования.  
Это фиксируется как осознанный риск.

## 30. FAILURE SCENARIOS

### 30.1. Импорт
Если импорт падает:

- боевые сущности не должны оказаться частично изменёнными;
- черновики могут сохраниться отдельно;
- пользователь должен видеть понятный статус ошибки;
- событие должно быть залогировано.

### 30.2. Восстановление
Если восстановление не завершилось:

- текущая рабочая копия не должна тихо исчезнуть;
- страховочная копия должна сохраняться;
- система должна показать диагностический статус;
- событие должно попасть в alert и audit.

### 30.3. Пересчёт статусов
Если пересчёт не завершился:

- операция не должна считаться полностью успешной;
- пользователь должен получить диагностическое уведомление;
- должен быть доступен повторный сервисный пересчёт.

### 30.4. Почта
Если письмо не отправлено:

- отчёт не должен считаться доставленным;
- система должна фиксировать каждую попытку;
- должна быть возможность ручной отправки.

---

## 31. НОВОСТИ И НПА

### 31.1. Назначение
Модуль служит только для информирования.

### 31.2. Ограничения
Он не должен:

- менять внутренние данные предприятия;
- автоматически создавать кадровые или ОТ-сущности;
- принимать юридически значимые решения за пользователя.

### 31.3. Требования к источникам
Источники должны быть:

- доверенными;
- явно перечисленными в настройках;
- отключаемыми;
- проверяемыми на доступность.

### 31.4. Логика обработки
Система должна уметь:

- кэшировать материалы;
- исключать дубли;
- помечать прочитанное;
- фиксировать дату последней проверки;
- не ломать основной контур при сбое источника.

## 33. ТЕХНИЧЕСКИЕ ОГРАНИЧЕНИЯ ДЛЯ AI-СБОРЩИКА

1. Не строить проект как web-first без отдельного согласования.
2. Не вводить обязательный сервер.
3. Не ломать модель одного главного оператора.
4. Не внедрять локальную сеть как обязательное условие.
5. Не писать бизнес-логику в UI.
6. Не давать импорту писать напрямую в боевые сущности.
7. Не давать внешнему НПА-модулю менять внутренние данные.
8. Не делать универсальный мастер-пароль.
9. Не отправлять пароль по почте.
10. Не считать закрытые наряды редактируемыми.
11. Не переиспользовать архивные карточки сотрудников.
12. Не делать массовую выдачу СИЗ на первом этапе.
13. Не использовать непонятные аббревиатуры в интерфейсе без расшифровок.
14. Не превращать медблок в медкарту.
15. Не перегружать главный экран всеми блоками сразу.
16. Не превращать оркестратор в god-object.
17. Не хранить вычисляемые статусы как единственный источник истины.
18. Не обходить audit и security-события в критичных сценариях.

## 34. ЗАПРЕЩЁННЫЕ УПРОЩЕНИЯ

Без отдельного согласования запрещено:

- выкидывать оркестратор;
- выкидывать recovery-механику;
- заменять сервисный код мастер-паролем;
- снимать разделение внутренних и внешних контуров;
- удалять историю закрытых нарядов;
- делать импорт в один клик без проверки;
- убирать страховочную копию перед восстановлением;
- убирать Фокус дня;
- убирать дерево подразделений с главной;
- превращать все реестры в один универсальный шаблон;
- добавлять диагнозы в медицинский блок;
- делать дрожание для предупреждений;
- делать навязчивую постоянную анимацию;
- позволять внешним источникам влиять на внутренний статус сотрудника напрямую.

## 35. ОТКРЫТЫЕ РИСКИ

1. Экспорт и бэкап без шифрования в первой версии являются осознанным риском.
2. Поддержка Excel и JSON не означает поддержку произвольных форматов без нормализации.
3. Реальный UI почти наверняка потребует нескольких итераций по скринам.
4. Дерево подразделений на больших объёмах может потребовать оптимизации.
5. Пороговые значения предупреждений будут уточняться в эксплуатации.
6. Поздний модуль шаблонов и документов пока не детализирован.
7. Recovery/service-code механизм требует аккуратной криптографической и процедурной реализации.
8. Внешние источники НПА могут менять структуру и ухудшать стабильность парсинга.

## 36. КРИТЕРИИ ГОТОВНОСТИ ПЕРВОЙ РАБОЧЕЙ ВЕРСИИ

Первая рабочая версия считается готовой, если:

- существует локальная установка;
- работает вход по паролю;
- работает восстановление доступа;
- работает главный экран;
- работают сотрудники и карточки;
- работают Инструктажи, СИЗ, Медицина, Наряды-допуски;
- работает движок статусов;
- работают уведомления и Фокус дня;
- работает полный импорт через черновики;
- работает полный экспорт;
- работает ручной и авто-бэкап;
- работает восстановление;
- работает ежедневный отчёт;
- работает чтение НПА/новостей;
- критические события логируются;
- права инспектора и руководителя разведены;
- ключевые failure-сценарии протестированы;
- нет прямой бизнес-логики в UI;
- нет обхода оркестратора в критичных операциях.

Это не журнал и не набор таблиц.  
Это локальная модульная desktop-система управления охраной труда, где:

- главный оператор — инспектор;
- ядро координации — оркестратор;
- доменные правила живут в профильных модулях;
- данные предприятия локальны;
- внешний контур изолирован;
- каждый доменный блок имеет свои правила критичности;
- UI строится как современный рабочий пульт без визуального шума;
- безопасность, восстановление доступа, audit, failure-сценарии и тестируемость обязательны с первого рабочего контура.

Система должна собираться сверху вниз:

`каркас -> сотрудники -> статусный движок -> доменные модули -> импорт/экспорт -> бэкап -> безопасность -> НПА -> логи -> UI-шлифовка`
